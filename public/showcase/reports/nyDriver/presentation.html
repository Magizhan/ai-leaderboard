3. Root Causes

Deep codebase analysis of top pain points

Root Cause: "Not Getting Rides" (20.8%)

How allocation works

  • Marketplace model — rides broadcast to pool, drivers bid (vs Ola/Uber dispatch)
  • Intelligent pool scores on 6 factors: acceptance rate, cancellation, speed, proximity, activity, ride frequency
  • GoHome pool gets priority in early batches
  • New drivers lack history → placed in random pool

Hidden root causes

  • Silent ride blocking — outstanding dues > maxCreditLimit blocks rides with NO notification
  • Rich-get-richer: high-frequency drivers scored higher → get more rides → even higher scores
  • Location staleness (180s filter) drops drivers with poor GPS
  • No proximity weight floor — farther drivers with better scores outrank closer ones

Key file: SendSearchRequestToDrivers/Handle/Internal/DriverPool.hs (lines 772-795)

Root Cause: Earnings Opacity (11.3%)

The transparency gap

  • FareBreakupPopup shows only 3 lines: estimated fare, "extra charges" (opaque delta), toll
  • Tips exist in API but are never shown to drivers
  • Parking charges in data model but never displayed
  • Ride history shows total only — no toll/tip/special charge indicators

Backend issues

  • 8+ open GitHub issues for fare calculation bugs across cities
  • 7+ issues for payment button infinite-loading
  • Special zone fee breakup mismatch (#4790)
  • USE_DUMMY_PAYOUT_HISTORY = true — settlement history shows dummy data
This is primarily a transparency problem, not a fare structure problem. NammaYatri's commission-free model actually gives drivers more per-ride than competitors — but drivers can't see this.

Key Takeaway

NammaYatri's driver app has strong ratings (4.7/5) and a powerful economic proposition — but driver trust is eroding. The good news: the top complaints are primarily transparency and UX problems, not fundamental business model issues. 36 frontend fixes across 4 sprints can meaningfully address 80% of complaints — with 5 already implemented on git branches.
