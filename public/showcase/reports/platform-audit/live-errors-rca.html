Critical Gaps

8 Critical Gaps Found

Security, data integrity, and reliability gaps requiring immediate action

GAP 1: Credential Leak in Logs

P0 SECURITY

Gupshup SMS API passwords and user phone numbers logged in plain text via OpenSearch. Full URLs with query params containing credentials exposed to anyone with log access.

GAP 2: Redis HMGET Bug

P0 BUG

Wrong argument count for HMGET breaks ALL multimodal/bus search. PayoutConfig decode failures: 10K+ errors/day — "expected Object, got null" across 7 city IDs.

GAP 3: No Booking Idempotency

P0 CRITICAL

Duplicate bookings possible on client retry. No idempotency key, no unique constraint on estimate_id. Payment retries can create duplicate payment intents via Juspay.

GAP 4: No Distributed Tracing

HIGH

No OpenTelemetry. Request IDs may exist but not propagated across services, Kafka messages, or logs. Manual debugging across 20+ services.

GAP 5: Zero Test Coverage in CI

HIGH

No tests running in CI. Critical path code (booking, payment, allocation, fare calc) has no automated test coverage. No regression safety net.

GAP 6: Drainer Has No Resilience

HIGH

Redis-to-PostgreSQL drainer has no retry, no backpressure, no dead letter queue. If it fails, data is silently lost. No alerting on drainer lag or crashes.

GAP 7: No PgBouncer

HIGH

10-25 PG connections per service, 79+66 pods = potential 3,000+ direct DB connections. No connection pooling layer. PostgreSQL max_connections at risk during scaling.

GAP 8: Cascading Failures

HIGH

BPP internal API errors cascade as E500 to riders (10K+/day). Nandi GTFS stop mapping breaks entire booking list. WhatsApp opt-in generates 10K+ spurious errors/day.
