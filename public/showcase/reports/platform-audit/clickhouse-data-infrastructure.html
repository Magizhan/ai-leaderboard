Streaming

Kafka → ClickHouse Ingestion Pipeline

4 ingestion patterns, 407 Kafka consumers, 30.4 billion messages consumed

Ingestion Patterns

1
CDC via Kafka (CollapsingMergeTree)
PostgreSQL → Debezium → Kafka → Queue table → Main table. Uses sign=1/-1 for insert/delete. Core ride-hailing tables.
2
Kafka → MV → Main Table
Raw Kafka strings parsed by materialized views into typed columns. Used for driver_location_updates, rider GPS, GPS raw data.
3
Direct Kafka Ingestion
No MV transformation. driver_eda_kafka (40.7B rows) and amnex_direct_data (1.2B) read directly from Kafka topics.
4
MV Aggregation (app_monitor)
Raw events → SummingMergeTree aggregates (per-minute counts, per-user counts, TDigest latency sketches).

What Needs Attention

60% of ClickHouse cluster + 100% of Kafka brokers invisible to monitoring
10 systemic data quality issues (boolean casing, \N strings, dirty enums)
48% of driver platform fees are PENDING or OVERDUE
76% of SOS safety events remain unresolved
OpenSearch is 93% Istio logs — app logs get only 2-day retention

The data infrastructure is fundamentally sound — ClickHouse ingests billions of rows reliably via Kafka CDC. The gaps are in monitoring visibility (exporters down), data hygiene (boolean/null inconsistencies), and operational alerting (fee collection, SOS resolution, recon pipelines). Fixing the P0 monitoring issues should be the immediate priority.
