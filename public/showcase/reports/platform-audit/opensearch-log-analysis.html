Architecture

OpenSearch Cluster Architecture

AWS OpenSearch cluster beckn-els — primarily an operational logs cluster

Cluster Health

GREEN
All Shards Assigned

12 nodes (9 data + 3 dedicated masters). 2,760 active shards (1,380 primary + 1,380 replica). Zero unassigned shards, zero pending tasks.

Role in Platform

Logs
Operational Infrastructure

Unlike ClickHouse (business data: rides, bookings, estimates), OpenSearch stores operational/infrastructure logs. Istio mesh logs, app logs, GPS metrics, payment logs.

Key Concern

Unparsed
Schema Not Optimized

All index families use minimal schemas (3-5 text fields). Rich structured data inside message/log fields is never parsed into indexed fields, making search extremely slow.

93% of storage (894 GB) is consumed by Istio service mesh access logs stored as raw text. Application logs get only 6.5% of storage with just 2-day retention.

The DriverTier Null Crisis

This error appears across three services (BPP, allocator, app-backend) at very high frequency. The allocator logs it per-driver-per-allocation-cycle.

Estimated: millions of errors per day

Search & Queries

Search & Query Patterns

Unstructured schemas make every query a full-text scan of millions of documents

beckn-lp-logs: The Most Valuable Index

Schema: Only 3 fields — @timestamp, message (JSON blob), tag

Volume: ~50M docs/day, all Haskell services in a single daily index

Query times: 5+ second took times observed for simple searches

No service_name, log_level, request_id, or error_code as separate indexed fields. Every query parses every document.

Istio Proxy: 690M Docs/Day Unsearchable

Envoy access logs stored as a single log text field. Cannot efficiently query:

• All 5xx errors to a service
• p95 latency by endpoint
• Traffic flow between services
• Client version error rates

What Fields Should Be Indexed

beckn-lp-logs parsing target:

message JSON → service_name, log_level, request_id, session_id, error_code, cluster

istio-proxy parsing target:

log text → http_method, request_path, http_status, duration_ms, response_flags, upstream_service, client_ip

atlas-sdk-logs parsing target:

message JSON → session_id, event_type, app_name, app_version, city_name, api_url, api_status

A single Fluentd/FluentBit parsing pipeline extracting structured fields at ingestion would unlock all 6 missing dashboards and reduce query times from 5+ seconds to sub-second.

Settlement cron logs (financial operations) expire in 48 hours. If a payout fails or is disputed, logs are gone. Regulators typically require 90+ day audit trails.
Reducing istio-proxy to 1 day or sampling at 10% would free ~300 GB — enough to extend beckn-lp-logs to 14 days, sdk-logs to 7 days, and settlement to 90 days.
