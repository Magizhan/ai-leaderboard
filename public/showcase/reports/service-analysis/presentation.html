Worst Endpoint

/v2/rideBooking/list/ — 54.7 RPS of 401s. Rider app polls with stale tokens; on 401 it factory-resets the app and logs user out.

Location Tracking: 65 err/s

Drivers sending GPS updates too frequently hit the sliding window rate limiter. 429 responses are expected behavior, not bugs — but inflate error counts.

Fix Plan

The Fix Series: 90.76% → 99.2%

Fixes #1 + #2 alone eliminate 68% of all errors in ~4.5 days

Fix #1: INVALID_TOKEN

37% of errors

56.7 err/s — ~3 days effort (client+server)

Add silent re-auth on 401 (client). Token expiry grace period for read-only endpoints (server). Stop background polling when app is backgrounded.

Alone raises availability to ~95%

Fix #2: RIDE_NOT_SERVICEABLE

31% of errors

48.4 err/s — ~1.5 days effort

Return empty driver list with serviceable: false instead of 400 error. Cache serviceability polygon client-side to prevent the request entirely.

Combined with #1: reaches 99.2%

Fix #3: LTS Rate Limiting

65 err/s

65 err/s — ~0.5 days (config change)

Increase rate limit or batch driver location updates client-side. These are expected 429s, not real failures — consider excluding from availability metrics.

Fix #4: RIDE_DOES_NOT_EXIST

23 err/s

23 err/s — ~1 day effort

Return empty location instead of throwing error (server). Stop polling driver location after ride completion (client).

Fix #5: Race Conditions (BPP)

Expected Behavior

RIDE_REQUEST_ALREADY_ACCEPTED (6.7/s) + 409 conflicts (1.0/s). Inherent to competitive ride acceptance auction model. Filter from dashboards — not bugs.

Fixes #6-8: Minor

Low Priority

GTFS 404s (missing stop codes in data), dashboard 400s (low traffic, outsized % impact), Prometheus /metrics 404s (scraping non-existent endpoint).
