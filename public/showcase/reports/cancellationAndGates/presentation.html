The Cancellation Problem

Why Rides Get Cancelled

Only 60.1% of requests receive driver responses in dense cities. 39.9% go unfulfilled due to supply-demand mismatch.
  • Pickup too far — drivers cancel when actual driving distance exceeds expectations
  • Destination undesirable — office parks (mornings), residential outskirts (evenings)
  • Confusing pickup points — no gate, wrong gate, blocked gate at airports/malls
  • Long wait times — rider cancels when ETA is too high or driver not moving
  • Supply hoarding — drivers in popular areas cherry-pick, ignoring peripheral requests

Geographic Clustering

Cancellations are not random — they cluster at specific locations where structural problems repeat.

Key Insight

Airports, metro stations, malls, and corporate parks generate disproportionate cancellations. These locations need structured pickup zones with gates — but only if placed using data, not intuition.

Attribution Capability

Strong Driver cancellations — GPS + distance + reason code available

Weak Rider cancellations — no GPS captured at cancel time

Automated Discovery

Uber/Grab use DBSCAN + KDE to detect cancellation hotspots. DiDi patented automated zone creation → deployment → updating lifecycle.

FIFO Queue Fairness

Grab KLIA: queue position updates every 5 min, 25-min return grace period. Uber/Lyft: digital FIFO with visible position and wait-time display.

Dynamic Zones

DiDi: time-restricted zones, trigger-based activation, auto-resizing via geohash precision. Grab: flight tracking for dynamic airport pickup adjustment.

Backend Audit Findings

Architecture: Two Parallel Systems

System A: lib/special-zone (Production)

Drives all ride flow — fare products, driver allocation, OTP verification. PostGIS MultiPolygon zones + gate_info with queue polygons, extra fare, entry fees, walk descriptions.

System B: app/special-zone (Standalone)

Simpler model, only system enforcing non-overlapping geometries via ST_Relate. Not used in ride flow.

Ride Flow Integration

  • Pickup zone check via ST_Contains at search time
  • Gate match enables SpecialZoneQueuePool (batch -1 priority)
  • Zone-specific FareProduct via Area ADT
  • specialLocationTag propagates through booking → fare calc
  • Drop zone re-check and fare recalculation at ride end

Mobile App Audit

Rider Experience

CLOSED ZONES (Airports)

Must select from predefined gates. Horizontal scroll list, auto-zoom to zone polygon. Nearest gate auto-selected via haversine.

OPEN ZONES (Malls)

Custom pickup allowed if >20m from any gate. Gate sub-polygons highlight on selection (orange selected, dashed unselected).

CANCELLATION UI

Dynamic reasons from server. No zone-specific reasons exist. No GPS captured at cancel time. Reason stage tracked (OnSearch/OnConfirm/OnAssign).

NammaYatri vs. Industry

What we do well, and where we fall short.

  • Metro Warrior specialization — unique in industry

  • PostGIS spatial queries — production-grade infrastructure

Proposed Solutions & Roadmap

P0 — Data Foundation
• Cancellation heatmap in control center (H3 overlay on zone map)
• Zone effectiveness dashboard (per-zone cancel rate, ride volume, completion rate)
• No schema changes needed — pure query + UI
P1 — Scalability
• Bulk zone import UI (CSV/KML upload to existing batch API)
• HDBSCAN hotspot detection (nightly batch)
• Auto-suggest zones with one-click creation
• Alpha shape boundaries + MCLP gate placement
P2 — Data Enrichment
• Capture rider GPS at cancel (2 nullable columns)
• Zone-specific cancel reasons ("gate closed," "wrong gate," "walking too far")
• Crowdsource API for real-time gate status
P3 — Advanced
• Dynamic zone lifecycle (time-based activation, auto-deprecation)
• FIFO queue visibility (position + wait time at gates)
• Switchback experiments for zone A/B testing

Validation Path

1. Prove geographic clustering (HDBSCAN + Getis-Ord Gi*)
2. Characterize hotspots (reason codes, time patterns)
3. Measure existing zone effectiveness
4. Pilot 3 new zones with diff-in-diff measurement

Success Criteria

≥3 significant clusters not covered by existing zones. ≥5% cancellation rate reduction with p<0.05. Measurably lower cancel rate inside zones vs. comparable outside locations.

Core Hypothesis

Cancellations cluster geographically, and zones/gates placed at high-cancellation locations — using data-driven boundaries and optimized gate placement — will measurably reduce them.
